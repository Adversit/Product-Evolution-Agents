## 1. 产品定位（Demo 版）

EvoPM Agent：面向产品迭代的多智能体需求决策系统。输入真实用户反馈与 GitHub Issues，自动完成：

```text
过滤 → 分类 → 查重 → 聚类 → 竞品/技术调研 → 需求质量门禁 → 机会评分 → 路线图 → 代码影响面 → 实现计划
```

全程证据可追溯、关键结论人工确认。

### 两条核心设计原则

1. **人工介入分级**：简单结论人工过目即可；复杂或涉及核心代码的，系统只提供建议，**永不直接修改代码**，必须人工自行执行。
2. **需求漏斗必须有出口**：不是所有反馈都转需求、不是所有需求都值得做。过滤、分类、与历史需求去重、质量门禁、优先级分流，每一级都有「淘汰/分流」出口。Demo 要刻意展示被滤掉的反馈和被分流的需求。

### 硬约束

| 项 | 约束 |
|---|---|
| 编排 | LangGraph（Python） |
| LLM | 智谱 GLM API：主链 `glm-5.1`，开发调试 `glm-4.7-flash`（免费） |
| 形态 | 后端 pipeline + CLI + Markdown 报告；前端仅在时间富余时做 |
| 数据策略 | 先 mock 数据跑通全流程，再接真实数据（GitHub API / web_search）测试 |
| 密钥 | `.env` 管理，纳入 `.gitignore`，提交 `.env.example` |
| 资源 | 黑客松 2 天 / 1 人（含增强项约 3 天工作量，按保底层→增强层分层，任意时刻可演示） |

## 3. 模块范围（17 模块逐项决定）

| 模块 | Demo 做什么 | 不做什么 |
|---|---|---|
| **M0 项目配置** | `product.yaml` + `evopm init` CLI 交互式创建 | 多项目管理、分析粒度多选（固定为单功能模块） |
| **M1 反馈导入** | CSV + 真实 GitHub API 拉 issues（mock 兜底）→ 统一 SignalItem；LLM 批量标注：12 类分类 / 情绪 / 模块归属 / **可行动性 6 档**（信息充分/信息不足/疑似重复/疑似误用/情绪表达/真实问题） | 其余 11 种反馈来源、人工状态流转、来源可信度分级 |
| **M2 外部源配置** | yaml：竞品列表（数量可配）+ 技术调研关键词 + mock 材料路径 | 信息源类型管理、启用/禁用、对标维度配置 |
| **M3 归一化** | 纯 Python 解析 CSV/JSON + M1 的 LLM 标注顺带完成字段抽取；保留**数据质量标注**字段（完整/部分缺失/噪声较高） | 独立清洗管线、证据组归并（由 M4 聚类覆盖） |
| **M4 发现聚类** | LLM 直接聚类（2–4 簇）；**12 类全量分类**；状态 4 档（新/已知/重复/信息不足）；每簇：标题/摘要/严重度/频次/代表性证据 id/候选需求/用户故事初稿；低质量反馈生成追问建议；**候选需求与历史需求池查重 → Duplicate 合并** | embedding 聚类、人工合并/拆分/重命名簇 |
| **M5 竞品调研** | 围绕选中簇自动生成 3–5 个调研问题 → GLM web_search + mock 兜底 → 结论三分类（可借鉴/不建议跟随/继续观察）+ 来源 URL + 证据强度 | 竞品评论挖掘、人工修正竞品判断（统一走最终评审） |
| **M6 技术调研** | 关键词生成 → web_search + 兜底 → **5 档成熟度**（成熟可用/可参考/需验证/高风险/不适合）+ 与需求匹配度 + 「为什么支持该需求」解释 | 论文检索 |
| **M7 需求草稿** | 背景/目标用户/痛点/功能范围/**非目标范围**/边界条件/待澄清问题/用户故事（角色+场景+收益）/证据映射；**PRD 草稿独立导出**为 .md 工件 | 非功能需求独立章节（并入边界条件）、历史 roadmap 查重（由历史需求池查重覆盖） |
| **M8 质量门禁** | **10 维评分**（清晰性/完整性/可测试性/验收标准明确度/证据充分性/范围控制/可实现性/一致性/用户价值明确度/产品阶段匹配度），每维 0–100 + 锚点描述；**门禁判定用代码规则**（总分 ≥70 且无 blocker → PASS）；enrich 用调研证据自动补全一轮后重评；quality_history 记录前后对比（Demo 高光：58→86）；门禁 3 出口：进方案设计 / 需补充信息 / 转文档客服 | 14 维全量、按问题类型差异化门禁、人工覆盖评分 |
| **M9 机会评分** | **10 维评分**（痛点频次/严重度/竞品差距/技术可行性/需求质量/实现成本/商业价值/战略适配度/时效性/是否影响核心路径）；**7 级优先级全量**（P0/P1/P2/P3/Support/Research/Duplicate）；Now/Next/Later；4 种特殊类型标注（高频低价值/低频高严重/竞品已解决我缺失/技术可行证据不足）；**权重写进 yaml**（团队偏好→权重映射） | 权重 UI 调整、人工拖拽改优先级 |
| **M10 方案设计** | 方案摘要/功能范围/非目标/核心用户流程/验收标准（功能性+非功能性，每条绑证据）/异常场景/测试场景/风险与依赖；**4 角色说明全量**（产品/研发/测试/客服运营） | 产品交互建议、独立任务拆解（移 M12 统一做） |
| **M11 代码影响面** | 输入真实 RAGFlow 目录快照 repo_map；模块级定位；**三档影响**（确定/可能/不确定，不确定项强制进待确认清单）；7 类影响对象（前端/后端接口/服务逻辑/数据模型/配置/测试/文档）；修改风险/建议修改顺序/验证点；**涉及核心模块清单的自动升为高风险** | 克隆并解析真实代码内容、文件行级定位 |
| **M12 执行计划** | 开发任务卡（**7 类任务类型**：产品/前端/后端/数据/测试/文档/运营客服）/修改建议/测试建议/implementation plan（有序步骤，每步含改动模块+验证点+风险）/**Changelog 草稿**；**门禁前置**（M8 gate 不过则拒绝生成执行计划） | **Patch 草稿、PR 草稿**（系统永不产出代码改动）、文档更新建议（降为文档类任务卡） |
| **M13 证据链审查** | `evidence_refs` 贯穿全部 schema + **每节点输出后代码校验引用合法性**（防 LLM 编造证据）；Critic Agent：证据强度 **5 档全量**（强/中/弱/无直接证据/仅模型推断）/过度推断标注/弱证据结论降权为观察项/生成待人工确认清单；**自动回炉一轮**（redo_target 指定节点，计数限 1 次，复审仍有问题则只标注交人工） | 证据卡独立存储层（报告渲染时现场生成） |
| **M14 报告导出** | **4 份 Markdown**：产品机会报告（评审版）/研发执行报告/PRD 草稿/管理层一页摘要 + state dump JSON；未确认结论自动加 `[未确认]` 标记 | PDF、报告模板选择、按范围筛选导出 |
| **M15 状态流转** | **12 状态枚举全量定义**（New Signal→…→Merged as Duplicate，demo 流程实际走其中约 6 个）；状态变化记录；Now/Next/Later 报告 section | 独立看板、公开 roadmap 摘要、反馈 addressed 联动 |
| **M16 评审工作台** | 3 个 CLI interrupt 点（①选问题簇 ②门禁兜底澄清 ③最终评审）；**5 种操作**：接受/驳回/编辑/要求重新分析/**要求补充证据**（重跑调研节点）；每操作记录理由；报告标注已确认/未确认 | 确认类型分类管理、评审前后差异视图 |

## 4. 人工介入分级策略（核心原则一）

| 风险级 | 范围 | 处理方式 |
|---|---|---|
| **低** | 反馈分类、情绪标注、聚类命名、摘要类结论 | 自动通过，报告中可见，评审工作台默认折叠、可抽查 |
| **中** | 门禁判定、优先级、竞品/技术结论、需求草稿内容 | 进入待确认清单，人工逐项：接受/驳回/编辑/补证据/重新分析 |
| **高** | 「不确定」影响面、修改建议、implementation plan、**凡涉及核心模块清单**（`rag/nlp`、`deepdoc/parser` 等） | **强制人工确认**；系统仅输出建议文档，不产出任何代码改动（无 Patch/PR），执行由人工在自己环境完成 |

## 5. 需求漏斗（核心原则二）

```text
25+ 条信号
 → 可行动性过滤（情绪表达/疑似误用/信息不足 → 滤出，生成追问建议）
 → 12 类分类 + 信号查重
 → 聚类成 2–4 簇 → 候选需求
 → 与历史需求池查重（重复 → Duplicate 合并，不再推进）
 → 质量门禁（不达标 → 自动补全一轮；仍不达标 → 人工澄清 / 转文档客服）
 → 机会评分（P3/Support/Research/Duplicate → 分流出口，不进执行）
 → 仅 P0/P1 进入方案设计与执行计划
```

Demo 数据刻意包含：2–3 条情绪/误用反馈（被滤出）、1 条与历史需求重复的候选（被合并）、1 个低优先级簇（被分流到 Later/Support）。报告需展示漏斗各级的淘汰记录。

## 6. LangGraph 编排

```text
START → intake（M0-M3 / Intake Agent）
  → discovery（M4 / Discovery Agent，含历史需求查重）
  → [interrupt① 人工选簇]
  → competitor_research ∥ tech_research（M5/M6 / Research Agent 双模式，fan-out/fan-in 并行）
  → quality_gate（M7+M8 / Requirement Agent）
       ├─ FAIL 且未补全 → enrich（用调研证据补全）→ 回 quality_gate
       ├─ FAIL 且已补全 → [interrupt② 人工澄清/强制放行/转文档客服]
       └─ PASS → opportunity（M9）→ solution_design（M10）（Strategy Agent）
  → engineering（M11+M12 / Engineering Agent）
  → critic（M13 / Critic Agent）
       └─ 严重问题且未回炉过 → 回炉指定节点（限 1 次）→ 复审
  → [interrupt③ 最终评审：5 种操作；「补充证据」→ 重跑调研节点]
  → report（M14 / 纯 Jinja2 渲染，无 LLM）→ END
```

- 门禁与回炉的循环计数在 state 中，硬性防死循环。
- interrupt 用 LangGraph `interrupt()` + `Command(resume)` + `MemorySaver`（单进程内存，不引入 sqlite）。
- 全链约 10–12 次 LLM 调用。

## 7. Agent 设计（A1–A12 → 7 个）

| Agent | 吸收角色 | 要点 |
|---|---|---|
| **Intake** | A1+A2+A3 | GitHub API 拉取 + 批量标注（12 类/情绪/可行动性 6 档/数据质量/疑似重复） |
| **Discovery** | A6 | 聚类 + 候选需求 + 用户故事初稿 + 历史需求查重；证据 id 闭包代码校验 |
| **Research** | A4+A5 | 双模式（竞品/技术）并行；GLM web_search，失败自动降级 mock；「只基于材料下结论，不足标 weak」 |
| **Requirement** | A7 | draft_and_score（10 维+锚点）+ enrich（只允许基于给定证据补全，标注来源） |
| **Strategy** | A8+A9 | 机会评分（yaml 权重注入）→ 方案设计（4 角色说明），两节点顺序调用 |
| **Engineering** | A10+A11 | 影响面三档 + 核心模块升高风险 + 任务卡/测试建议/impl plan/Changelog；**不输出 diff** |
| **Critic** | A12 | 5 档证据强度 + 过度推断 + 降权 + 待确认清单 + 回炉指令（redo_target） |

## 8. 技术约束与稳定性兜底

- **定位：黑客松 Demo** —— 无鉴权、无多租户、无分发部署、无计费、无企业权限。
- **稳定性兜底必须有**：
  - LLM 429/超时 → 指数退避重试；
  - 结构化输出校验失败 → 自动重试一次；
  - LLM 响应磁盘缓存（prompt hash → json）+ `--replay` 离线重放（对抗现场断网）；
  - 外部依赖（GitHub API / web_search）失败 → 自动降级 mock；
  - 证据引用代码侧校验。
- **GLM 集成**：`langchain_openai.ChatOpenAI(base_url="https://open.bigmodel.cn/api/paas/v4/")`；结构化输出 `with_structured_output(method="function_calling")`（**不可用 json_schema 模式**）；temperature ∈ (0,1)，取 0.1（**不能取 0**）；主链 `glm-5.1`，开发期 `glm-4.7-flash`（免费，并发低注意 429）。
- **web_search**：chat 请求 `extra_body` 传 `{"tools":[{"type":"web_search","web_search":{"enable":true,"search_engine":"search_pro","search_result":true}}]}`，返回的搜索结果（标题/链接/摘要/日期）直接作为证据来源。
- **密钥管理**：`ZHIPUAI_API_KEY`、`GITHUB_TOKEN`（可选）放 `.env`；提交 `.env.example`；`.gitignore` 覆盖 `.env`、`runs/`、`__pycache__`、LLM 缓存目录。
- **git**：每完成一个独立功能即 commit；模块开发使用 git worktree 并行分支。

## 9. 成功标准

1. `evopm run --replay` 离线端到端冒烟测试通过（确定性）。
2. GLM 真实跑通十步演示：导入（含真实 issues）→ 聚 3 簇（有反馈被滤出）→ 选簇 → 并行调研（web_search 带来源）→ 门禁 58→86 → P0（有簇被分流）→ 方案 → 影响面（核心模块标高风险）→ Critic 待确认 → 人工评审 5 操作 → 4 份报告。
3. 报告检查：每条结论 evidence_refs 可回溯到原文；驳回项标 `[未确认]`；漏斗各级淘汰记录可见。
4. 演示彩排 ≤3 分钟；现场断网时 `--replay` 可完整演示。
