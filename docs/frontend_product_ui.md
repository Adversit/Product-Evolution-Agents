# 前端约束（一）：产品级决策网页

**定位：** 把 EvoPM 的分析结果做成一个**「产品需求决策工作台」**——给产品经理/团队看的成品界面，重**审美与设计感、可信度、决策导向**。
**配套：** 接口见 `frontend_api.md`；内容样例见 `docs/sample_reports/`（glm-5.1 真实产出）；字段以 `schemas.py` 为准。
**与「过程演示版」的分工：** 本版**不展示**节点耗时、LLM 调用数、计数器、原始 JSON、流水线动画等工程内部信息——那些属于 `frontend_demo_ui.md`。本版只呈现**结论与证据**，干净、专业、可信。

---

## 1. 受众与目标

- **受众：** 产品经理、团队负责人、（演示时）评委中关注「产品价值/决策质量」的人。
- **核心目标：** 让人**30 秒内**从首页读懂「**问题 → 证据 → P0 建议 → 投入 → 风险**」，并相信这个结论是有据可查的。
- **一句话气质：** 像 Linear / Notion / Vercel 后台那样**克制、现代、有设计感**，不是后台表格堆砌，也不是炫技动画。

## 2. 设计原则（硬约束）

1. **克制专业**：中性底色 + **单一强调色**；大量留白；强排版（清晰的字号层级、行高）。不要彩虹色、不要重阴影、不要无意义动效。
2. **信息层级分明**：每屏一个主信息；次要信息收起/折叠。决策结论（P0、优先级、gate 通过）要**视觉最强**。
3. **数据可信感**：每条结论可**一键溯源到原文证据**（证据卡）；未确认/被驳回的结论显式标 `[未确认]` 并弱化；mock 来源标注清楚但不喧宾夺主。
4. **决策导向**：围绕「该不该做、做什么、投入多少、风险」组织，而非围绕「系统跑了哪些步骤」。
5. **一致性**：建立轻量设计系统（颜色 token、字号阶梯、间距栅格、圆角、卡片/标签/徽章组件）；深浅色二选一即可（建议浅色为主）。
6. **响应式**：优先桌面（演示用大屏）；窄屏可读即可，不强求移动端完美。

## 3. 核心视图（信息架构）

> 左侧导航 6 个区；顶部是产品/模块/运行模式的轻量上下文条。

1. **概览（Executive Dashboard）** — 默认首页
   - 执行摘要卡：问题一句话 + P0 建议一句话（来自 `executive_summary`）。
   - 漏斗 KPI：`N 信号 → 过滤 → 去重 → K 簇 → 1 焦点 → P0`（`/api/funnel`），做成精致的横向漏斗或 KPI 卡片组。
   - 焦点需求质量徽章：**58 → 86 PASS**（`quality_history` 首末），点进详情。
2. **问题与机会（Opportunities）**
   - 问题簇总览：卡片网格，DUPLICATE 簇打「已在路线图」标签并弱化（不可深挖）。
   - **焦点需求详情**（重点页）：标题/背景/痛点/目标用户/用户故事/验收标准/范围/非目标/边界；**质量评分 10 维雷达图 + 前后对比（初评 vs enrich 后）**；每条用户故事/验收标准可点 `evidence_refs` 弹证据卡。
3. **竞品与技术（Research）**
   - 竞品发现按 `verdict`（adopt/avoid/watch）分组的卡片；技术发现按 `maturity` 分组；每条标证据强度与来源（mock 标注但低调）。
4. **优先级与路线图（Roadmap）**
   - 机会评分表（10 维加权 + 总分 + 优先级 + horizon + 特殊类型）；
   - **Now / Next / Later 看板**（`roadmap`，焦点簇高亮）。
5. **研发执行（Engineering）**
   - 代码影响面地图：按 `impact_level` 分组，**核心模块 ⚠ 醒目**；
   - 任务卡（按 `type`）、实施计划（有序步骤含 verify）、Changelog 草稿、**风险与待确认清单**（`pending_confirmations`）。
6. **报告中心（Reports）**
   - 4 份报告（`/api/reports`）的精排阅读视图（Markdown 渲染，目录锚点）；支持复制/导出。

## 4. 关键交互

- **证据卡溯源**：任意 `evidence_refs` 标签 hover/click → 弹出 `/api/evidence/{id}`：原文摘录（≤120 字）+ 来源 + 强度。这是「可信度」的核心交互，必须顺滑。
- **质量前后对比**：雷达图叠加初评/终评两条；total 数字滚动从 58→86。
- **`[未确认]` 渲染**：`human_decisions` 中 `action=reject` 的对象，标题加 `[未确认]` 前缀并降饱和；`critic_review` 中 `demote_to_observation` 的移入「观察项」分区。
- **筛选/分组**：簇按 severity、竞品按 verdict、技术按 maturity、任务按 type。
- **导出**：报告导出 md（可选 pdf）。

## 5. 不做（避免跑偏）

- 不展示流水线节点动画、节点耗时、`llm_call_count`、`enrich_rounds` 等**工程内部指标**。
- 不暴露原始 `state.json` / 字段调试视图。
- 不做登录、多租户、设置页等与 demo 无关的页面。
- 不要为炫技加动画；动效仅用于「焦点切换、数字变化、证据弹层」等克制场景。

## 6. 验收标准

- [ ] 首页 30 秒可读懂「问题→证据→P0→投入→风险」。
- [ ] 焦点需求页能看到 10 维质量雷达的**前后对比**与 total 58→86。
- [ ] 任一结论上的 `evidence_refs` 能一键弹出原文证据卡。
- [ ] 核心模块改动有 ⚠ 视觉强调；被驳回结论显示 `[未确认]`。
- [ ] 4 份报告可精排阅读。
- [ ] 整体观感**有设计感、不像后台模板**；同行评价「像个真产品」。

## 7. 数据来源速查

| 视图 | 来源 |
|---|---|
| 执行摘要 / 漏斗 | `/api/reports/executive_summary`、`/api/funnel` |
| 簇 / 焦点需求 / 质量对比 | `/api/state`：`clusters`、`focus_candidate`、`quality_history` |
| 竞品 / 技术 | `/api/state`：`competitor_findings`、`tech_findings` |
| 机会 / 路线图 | `/api/state`：`opportunity`、`roadmap` |
| 代码影响 / 任务 / 风险 | `/api/state`：`code_impact`、`execution`、`critic_review` |
| 证据卡 | `/api/evidence/{ref_id}` |
| 报告全文 | `/api/reports/{name}` |
