## 录制前准备（务必先做）

1. **用 replay 模式**，全程离线、确定性、零等待，断网也能录：
   - CLI：`evopm run --replay`
   - 前端：两套都接 `glm-5.1` replay 数据，直接 `npm run dev`。
2. **压掉 CLI 顶部的 LangGraph msgpack 告警**（否则刷一屏红字很难看）：
   - PowerShell 录制前先设 `$env:PYTHONWARNINGS="ignore"`，再 `uv run evopm run --replay`。
3. 预热：录制前把两套前端各跑一遍，确认页面已加载、数据已就位。
4. 浏览器开 **全屏 / 隐藏书签栏**；终端字号调大到后排能看清。
5. 关掉所有通知、IM 弹窗。
6. 录屏建议 1080p、30fps，鼠标移动放慢，每切一屏停 1–2 秒给观众反应。

**主推路径＝前端 + CLI 混合**（视觉好又能看到真实执行）。若现场只能录一个窗口，用「流水线透视台」即可覆盖全链。

**画面：** README 顶部 hero 区（项目名 + 一句话 + 那张 Observatory GIF）。

**口播：**
> 大家好，我们的项目是 **EvoPM Agent —— 产品进化智能体**。
> 它解决一个所有产品团队每天都在做、却很难做好的事：**把成百上千条分散的用户反馈和 GitHub Issue，自动变成有证据支撑、可以直接排期的产品决策。**
> 它是一套基于 **LangGraph 的多智能体系统**，今天我用一条真实数据链路给大家完整演示。

## ② 痛点（0:25–1:00）

**画面：** 一页痛点字幕（用 PPT 第 3 页截图即可），或直接口播配反馈截图。

**口播：**
> 产品经理面对的现实是三个困境：
> **第一，淹没**——反馈太多，情绪宣泄、重复提交、用错功能混在一起，人工根本筛不过来；
> **第二，拍脑袋**——优先级靠感觉和嗓门，缺少证据，做完了不知道为什么做；
> **第三，断链**——需求、竞品、技术可行性、代码影响面各算各的，最后写出来的 PRD 没人能追溯它到底依据了哪条反馈。
> EvoPM 的核心理念就两条：**需求漏斗必须有出口**——不是所有反馈都值得做，每一级淘汰和分流都留痕；**所有结论必须有证据闭包**——每个判断都能回溯到真实的原始反馈。

---

## ③ 架构总览（1:00–1:35）

**画面：** PPT 的「技术架构」页（14 节点 DAG 图），或前端 Observatory 的全景 DAG。

**口播：**
> 系统是一条 **14 节点的 LangGraph 流水线**，由 **7 个智能体**协作：
> 信号导入与标注 → 聚类与历史查重 → **竞品 / 技术调研并行** → 需求质量门禁 → 机会评分 → 方案与执行计划 → **对抗式审查 Critic**。
> 关键设计有三个：
> 一是 **3 个人工介入断点**——选问题簇、门禁兜底、最终评审，风险分级，**系统只产出建议文档，永不直接改代码**；
> 二是 **门禁不达标会自动回炉**——用调研证据补全后重评；
> 三是 **全链稳定性兜底**——LLM 退避重试、结构化输出校验重试、磁盘缓存加离线重放、外部依赖失败自动降级。

## ④ 实跑 · 流水线透视台（1:35–2:55）★ 全片高潮

**画面：** 浏览器 `http://localhost:5173`。

**操作 & 口播：**

1. **（1:35）** 点 ▶ 播放，14 节点蛇形 DAG 依次点亮。
   > 这是真实的一次执行回放，数据产品是 **RAGFlow**，82.6k star 的开源 RAG 项目。我们导入了 **27 条**用户反馈和真实 GitHub Issue。

2. **（1:55）** 鼠标指向左侧/侧栏的漏斗计数。
   > 看漏斗：27 条信号，**可行动性过滤淘汰掉情绪宣泄和误用的，留下 5 条进入聚类**，聚成 **3 个问题簇**，其中**一个簇被历史需求查重判定为 Duplicate、直接合并出局**，最终深挖 **1 个焦点需求**。每一级淘汰都看得见。

3. **（2:20）** 停在 `quality_gate` 节点，指它的 10 维分数条。
   > 这里是质量门禁。需求初评 **61 分没过线**，系统**自动用调研证据补全了一轮**，重评跳到 **86 分**——这个 **61→86 的跃升**就是「自动回炉」在起作用。

4. **（2:40）** 指出并行分支和紫色虚线 ghost 路径。
   > 注意这两条**并行的竞品和技术调研**会 fan-in 汇到门禁；这些紫色虚线是**没有触发的条件分支**——澄清、重做、补证据——它们存在，只是这条数据没走到。流程是真分支，不是写死的。

5. **（2:50）** 点任意节点弹出检视抽屉，切换「结构化 ↔ 原始 JSON」。
   > 点开任意节点，能看到它的结构化输出字段，还能切到原始 JSON——**执行过程完全透明、可定位**。

## ⑤ 实跑 · 决策工作台（2:55–3:55）

**画面：** 浏览器 `http://localhost:5174`（浅色 Linear 风）。

**操作 & 口播：**

1. **（2:55）** 停在「焦点需求」首屏：大号 **86** 分 + 10 维雷达。
   > 切到给评委和团队看结论的决策工作台。焦点需求是「**解析任务稳定性修复——分阶段状态机、结构化错误透传与自动重试**」，质量总分 86，**机会总分 86.6，判定 P0、Now 阶段**。

2. **（3:15）** 点一个 `evidence` 角标，弹出证据弹层。
   > 点任意一条结论的证据角标——弹出**原文摘录、来源、证据强度**。这就是「**证据闭包**」：每个判断都能回溯到真实反馈，编造的引用会被代码层直接剔除。

3. **（3:30）** 左导航切到「问题簇总览」→ 指 DUPLICATE 标记 →「机会评分·路线图」Now/Next/Later。
   > 问题簇总览里能看到被标记 **DUPLICATE** 的那一簇；机会评分把需求分流到 **Now / Next / Later** 路线图，低价值的不会进执行。

4. **（3:42）** 切「研发执行」→ 指核心模块 ⚠ 高风险标记。
   > 研发执行这里，凡是命中 **核心模块清单**（比如 `deepdoc/parser`、`rag/nlp`）的改动**自动升为高风险、强制人工确认**。

5. **（3:50）** 切「报告中心」，快速划过 4 份报告。
   > 最终产出 **4 份 Markdown 报告**：产品机会、研发执行、PRD 草稿、管理层一页摘要——未确认的结论都带 `[未确认]` 标记。
