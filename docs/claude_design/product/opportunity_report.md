## 焦点需求

**解析任务稳定性修复——分阶段状态机、结构化错误透传与自动重试**（req-01，源自簇 clu-01）

- 背景：RAGFlow 用户在多种高频场景遭遇解析任务卡死或失败：80–100MB+ 大文件长时间转圈后直接失败（sig-001, sig-003）、GraphRAG 启用后任务队列堆积导致整体卡死（sig-013, sig-014）、docx 文件解析卡在 0%（sig-005）、PDF 解析完成后长期处于 pending 状态（sig-006, sig-007）。竞品调研显示 Dify、AnythingLLM、Open WebUI 同样面临大文件/批量场景的卡死问题且均未给出成熟方案（cf-02, cf-04, cf-06），说明这是行业共性痛点。根因分析（tf-05, tf-06）表明：当前解析流程将多步骤（提取→OCR→分块→嵌入→索引）压缩为单一「解析中」状态，worker 崩溃后状态机无法推进到 failed，任务永久卡死；同时失败后不记录结构化原因，错误链路在「异常捕获→存储→前端展示」环节断裂，用户无法获知具体失败原因，只能盲目重试或重新上传大文件。该问题出现频率极高（frequency=10）、严重度为 critical（sig-021, sig-022, sig-023），已确认为 known 状态。
- 目标用户：文档管理者 / 知识库管理员、使用 GraphRAG 的 RAG 应用开发者、处理大批量文档的企业级用户
- 痛点：用户上传 80–100MB 的大文件后，解析任务经常长时间转圈甚至直接失败（sig-001, sig-003）；失败时没有任何错误原因提示，只看到笼统的「解析失败」，不知道是格式不支持、OCR 超时、还是内存不足。GraphRAG 场景更严重：启用后任务队列堆积、worker 崩溃，任务永久卡在 pending/0%（sig-013, sig-014），用户只能反复重试或重新上传，每次重试都耗费大量时间和带宽。docx 卡在 0%（sig-005）、PDF 解析后长期 pending（sig-006）也频繁出现，导致知识库无法按时交付，严重影响日常工作流。
- 业务目标：将解析任务成功率提升至 95% 以上（含大文件场景），消除 GraphRAG 任务队列卡死问题；使每个失败任务都能给出结构化错误原因与可操作建议（如「OCR 超时，请重试」「格式不支持，请转换」），降低用户因解析卡死导致的流失率和工单量。

## 竞品对比（按 verdict 分组）

### ADOPT
- **Dify**（cf-01，强度 moderate）：Dify 的文档列表提供「处理中 / 可用 / 错误」三种状态标签，长文档批量处理时有进度提示，失败文档标红并支持重新处理。但材料未提及细粒度的步骤级进度（如'正在提取文本→正在分段→正在嵌入'的阶段提示）或具体错误码/错误原因描述，更多是文件级的状态展示。
  - 启示：Dify 的三态标签（处理中/可用/错误）+ 标红 + 重新处理是轻量但用户体感好的设计，RAGFlow 应至少对标此基础层：确保每个文档有明确的状态展示、失败时给出可操作的重新处理入口。步骤级进度可视化则是双方共同缺口，可作为 RAGFlow 的差异化突破点。
  - 差距：RAGFlow 在解析状态展示和失败反馈方面不如 Dify 直观，缺乏面向用户的清晰状态标签和一键重试入口。但 Dify 也仅做到文件级状态，步骤级进度双方均有缺口。
### AVOID
- **Dify**（cf-02，强度 moderate）：材料明确指出 Dify 在'超大文件或海量文档批处理时偶发处理缓慢、卡在处理中'，说明 Dify 同样面临大文件/批量场景的卡死问题。材料未提及 Dify 有文件大小上限、分片上传、流式解析或 OOM 防护机制，也未提及失败时给出具体错误原因。高质量索引（Q&A 模式、父子分段）还会显著增加 token 与时间成本，进一步加剧卡死风险。
  - 启示：大文件解析卡死是行业共性问题，Dify 尚未给出成熟方案，RAGFlow 不必跟随竞品做简单的状态展示修复，而应在大文件解析架构层面（如流式分块、内存控制、超时熔断）做差异化投入，这才是核心竞争壁垒。
  - 差距：Dify 在大文件处理上与 RAGFlow 面临同类问题，不构成差异化优势。
- **Open WebUI**（cf-04，强度 moderate）：材料明确指出 Open WebUI 的文档上传处理'较快但偏黑盒，没有面向用户的细粒度解析进度或步骤可视化'。用户在上传后无法获知解析处于哪个阶段、预计多久完成，也缺乏失败后的结构化反馈（错误码、原因描述）。其 RAG 能力定位为本地对话前端的附属功能，解析体验不是产品重点。
  - 启示：Open WebUI 在解析进度可视化方面是反面参照——用户上传后处于黑盒状态、无法感知进度或失败原因。RAGFlow 不应跟随这种设计，反而应将其作为'避免成为这样'的警示，优先补齐步骤级进度展示和结构化错误反馈。
  - 差距：Open WebUI 在解析进度可视化方面明显弱于 RAGFlow 需求预期，不构成可对标的能力。
- **AnythingLLM**（cf-06，强度 moderate）：材料明确指出 AnythingLLM 在'大文件或大批量文档嵌入耗时较长且进度反馈粗'，与 RAGFlow 面临同样的大文件性能瓶颈。其文档处理使用'统一的文本提取 + 分块管线，参数相对固定'，没有针对大文件的差异化处理机制。材料未提及文件大小上限、流式解析、OOM 防护或失败后的结构化错误反馈。
  - 启示：AnythingLLM 的大文件处理能力同样薄弱，且反馈更粗糙（'进度反馈粗'），说明大文件解析卡死是行业共性痛点而非 RAGFlow 独有缺陷。RAGFlow 应利用自身在深度解析上的技术积累，率先在此场景给出系统级解决方案（流式分块、内存上限、渐进式进度），形成竞争壁垒。
  - 差距：AnythingLLM 在大文件处理上与 RAGFlow 面临同类问题且反馈更粗，不构成差异化优势。
### WATCH
- **Dify**（cf-03，强度 no_direct）：提供的 Dify 调研材料完全未涉及 GraphRAG / 知识图谱构建功能或任务队列调度机制（并发控制、优先级队列、超时熔断、失败隔离等）。无法从材料判断 Dify 是否支持 GraphRAG，也无法判断其任务队列架构。
  - 启示：Dify 材料未覆盖 GraphRAG 与任务队列调度，无法形成有效对标。RAGFlow 的 GraphRAG 卡死问题需要在自身架构层面做设计决策（如独立 worker 池、并发上限、失败隔离），而非参考 Dify 的做法。
- **Open WebUI**（cf-05，强度 moderate）：材料提到 Open WebUI 支持接入外部内容提取引擎（如 Apache Tika、Docling）来增强对 PDF/Office 文档的解析，默认使用内置的简单文本提取管线。这种'可插拔解析引擎'的架构允许用户根据文档类型选择不同引擎，但材料未提及系统层面的自动 fallback（如主解析器失败后自动切换备用解析器）或预处理转换（如 docx→PDF 再解析）策略，更多依赖用户手动配置引擎选择。
  - 启示：Open WebUI 的可插拔引擎思路有参考价值：RAGFlow 可以考虑在自身深度解析引擎之外，提供 fallback 解析器接口（如内置引擎失败时自动降级到轻量提取器），形成多层容错机制。但当前优先级应低于核心卡死问题的修复。
  - 差距：RAGFlow 在深度文档解析（版面还原、表格提取）上有明显优势，但'可插拔解析引擎'的架构灵活性值得关注——Open WebUI 允许用户接入 Tika/Docling 作为替代或增强引擎。
- **AnythingLLM**（cf-07，强度 moderate）：材料指出 AnythingLLM 将上传与嵌入分为两步，界面会显示文档是否已嵌入以及文件级嵌入进度，但'缺少解析步骤级的细粒度可视化'。用户能看到'某文件正在嵌入'但无法得知当前解析处于哪个具体阶段、预计完成时间、或失败的具体原因。文件级进度有但步骤级缺失，与 RAGFlow 现状类似。
  - 启示：AnythingLLM 也只做到了文件级进度，步骤级可视化同样是行业空白。RAGFlow 若能率先实现解析步骤级进度（如'提取文本→识别版面→提取表格→语义分段→嵌入'的阶段展示），将在用户体验上形成差异化优势。这是投入产出比较高的方向。
  - 差距：RAGFlow 与 AnythingLLM 在解析步骤级可视化上处于同一水平线，双方均有缺失。
- **AnythingLLM**（cf-08，强度 weak）：材料描述 AnythingLLM 的上传与嵌入分两步执行，隐含一定的异步处理特征（先上传文档到文档管理器，再选择性嵌入到工作区），但未明确提及是否使用了独立 worker 池、任务状态机或消息队列（如 Celery、BullMQ）。无法从材料判断其解析任务是否与主进程隔离。
  - 启示：AnythingLLM 的'上传→嵌入'两步分离设计隐含了任务解耦思路，但材料不足以判断其架构细节。RAGFlow 在任务队列架构设计上需自行决策：采用独立 worker 池 + 状态机是业界处理此类问题的标准方向，可参考但不需依赖竞品验证。

## 技术方案（按成熟度）

### reference
- **解析状态机 + 心跳超时检测机制**（tf-01，话题：stability，强度 moderate）
  - 契合理由：调研问题 Q1/Q3 的核心方案：材料指出当前实现把多步解析（提取/OCR/分块/嵌入/索引）压缩成单一「解析中」态，无法区分「卡住」与「在跑」。引入分阶段状态枚举 + worker 心跳时间戳 + 超时阈值判定（超阈值无心跳→标记疑似卡住），可直接解决 GraphRAG 任务队列中 worker 崩溃后任务永久 pending 的问题。状态机是成熟工程实践，有大量参考实现可借鉴。
  - 成本：medium — 主要改动在 task_executor 侧拆分解析阶段并写入心跳/进度，需扩展状态表数据模型，但不需要引入新中间件 ｜ 风险：心跳超时阈值需调参：过短会误杀正常大文件解析（OCR/嵌入阶段耗时长），过长则失去卡死检测意义；需按文档类型和文件大小设置动态阈值
- **解析失败重试装饰器（指数退避 + 错误分类）**（tf-02，话题：stability，强度 moderate）
  - 契合理由：直接解决 Q4「失败后用户无反馈、无自动重试」：材料提出对瞬时错误（网络抖动、依赖 5xx、超时）做有限次指数退避自动重试，对确定性错误（格式不支持、文件损坏）直接落 failed + 原因码。这对 GraphRAG 场景下因嵌入/向量库抖动导致的间歇性失败尤为有效，避免用户反复手动重试。
  - 成本：low — 核心是给解析任务包一层错误分类 + 退避重试装饰器，不涉及架构级改动 ｜ 风险：重试必须幂等：若之前的解析已部分写入分块/索引，重试会造成脏数据或重复记录；需先实现分块/索引的 upsert 或事务回滚机制作为前置条件
- **结构化错误码 + 失败原因透传前端**（tf-03，话题：stability，强度 moderate）
  - 契合理由：解决 Q4「错误链路断裂」：材料指出当前失败后不记录结构化原因，错误链路在「异常捕获→存储→前端展示」环节断裂。建议每次失败写入结构化错误（错误类型、阶段、message、可重试标志），前端展示原因并提供手动重试入口。这能让用户看到「OCR 超时」「格式不支持」等具体原因而非笼统的「解析失败」，大幅减少无效重试。
  - 成本：medium — 需扩展状态数据模型存错误详情字段，并贯通 task_executor→后端 API→前端展示全链路 ｜ 风险：错误分类体系需要设计完整，覆盖格式不支持、OCR 超时、OOM、依赖服务抖动等多类错误；分类不准会导致可重试错误被跳过或不可重试错误浪费重试资源
- **分阶段进度条 + 步骤名展示**（tf-09，话题：ux，强度 moderate）

  - 契合理由：引用溯源属于检索/问答阶段的体验优化，与本簇「解析卡死/失败」无关。材料虽提到解析阶段需记录 chunk 定位信息（页码/偏移/bbox），但这是为引用溯源服务的，不能解决解析阶段的稳定性问题。在当前优先修复解析稳定性的语境下不适合纳入。
  - 成本：high — 需贯通解析-检索-前端全链路记录和透传定位信息，改动面广 ｜ 风险：与当前修复解析卡死的核心目标不匹配，分散开发资源；且改动面广，在系统不稳定时引入更大范围的变更有风险
- **多知识库检索结果归一化**（tf-13，话题：RAG 检索质量，强度 moderate）
  - 契合理由：多知识库分数归一化解决的是「多库检索时某库分数压制其他库导致漏召回」的问题，与解析任务卡死/失败完全无关。在当前簇中讨论此方案属于偏题。
  - 成本：medium — 需在检索打分层实现多库结果统一归一化逻辑 ｜ 风险：与当前需求完全不匹配，不应在解析稳定性修复的工期内纳入
### mature
- **混合检索（BM25+向量）+ reranker**（tf-12，话题：RAG 检索质量，强度 moderate）
  - 契合理由：混合检索与 rerank 是检索质量优化方案，属于 mature 能力。但与当前问题簇「解析卡死/失败」无直接关联——检索质量优化不能解决解析阶段的 worker 崩溃、OOM、超时等问题。仅当解析稳定性修复完成后再考虑检索质量提升时才有价值。
  - 成本：medium — 增加一次 reranker 模型推理，延迟与算力成本可通过限制重排候选数控制 ｜ 风险：在解析稳定性未解决前引入检索层变更，会增加系统整体复杂度和排障难度；核心模块改动需谨慎评估对既有召回的回归影响
