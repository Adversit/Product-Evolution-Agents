# 管理层摘要 — RAGFlow

> 模块：文件上传与问答质量 ｜ 阶段：growth ｜ 运行模式：`replay`

## 问题

解析任务稳定性修复——分阶段状态机、结构化错误透传与自动重试：用户上传 80–100MB 的大文件后，解析任务经常长时间转圈甚至直接失败（sig-001, sig-003）；失败时没有任何错误原因提示，只看到笼统的「解析失败」，不知道是格式不支持、OCR 超时、还是内存不足。GraphRAG 场景更严重：启用后任务队列堆积、worker 崩溃，任务永久卡在 pending/0%（sig-013, sig-014），用户只能反复重试或重新上传，每次重试都耗费大量时间和带宽。docx 卡在 0%（sig-005）、PDF 解析后长期 pending（sig-006）也频繁出现，导致知识库无法按时交付，严重影响日常工作流。

## 建议（P0 一句话）

**P0｜now**：解析稳定性是 RAG 产品核心链路入口，频次最高（10）+严重度 critical + 多场景覆盖，且大部分修复方案为成熟工程实践（状态机/心跳/错误分类/重试），应立即启动 Now 阶段修复。

## 投入预估

- 解析状态机 + 心跳超时检测机制：成本 medium — 主要改动在 task_executor 侧拆分解析阶段并写入心跳/进度，需扩展状态表数据模型，但不需要引入新中间件
- 解析失败重试装饰器（指数退避 + 错误分类）：成本 low — 核心是给解析任务包一层错误分类 + 退避重试装饰器，不涉及架构级改动
- 结构化错误码 + 失败原因透传前端：成本 medium — 需扩展状态数据模型存错误详情字段，并贯通 task_executor→后端 API→前端展示全链路
- 解析中间状态压缩为单一态——根因定位：成本 medium — 需要将 task_executor 内的解析流程拆分为独立可上报的阶段，每阶段写入当前步骤名和进度，改动集中在解析执行器层
- 失败后无结构化错误记录——掩盖真实问题：成本 low — 核心是在失败时捕获异常并分类写入错误字段（错误类型/阶段/message/可重试标志），改动范围在异常处理层
- 重试幂等性缺失风险——重复写入脏数据：成本 high — 需要实现分块/索引/图谱构建的幂等写入（upsert 或先删后建），GraphRAG 场景下幂等性设计更复杂，涉及实体去重和关系合并
- 任务卡 12 张，实施步骤 12 步。

## 风险

- 重试幂等性不完整导致脏数据：本期仅覆盖分块/索引 upsert 幂等，GraphRAG 场景下图谱构建（实体抽取/关系建立）的幂等性未解决（tf-07, tf-23 cost=high）。若 GraphRAG 任务触发重试可能产生重复实体节点和关系边，污染图谱质量。缓解措施：GraphRAG 任务的重试逻辑需额外增加「跳过已完成子图」的前置检查，或暂时对 GraphRAG 任务禁用自动重试仅保留手动重试 + 全量清理重建。
- 心跳超时阈值设置不当导致误杀正常任务：大文件 OCR 阶段单批次处理可能超过预期时间，若阈值过紧会误判正常任务为卡死并触发不必要的重试（浪费资源 + 延迟）。缓解措施：阈值设计基于 tf-01, tf-19 的参考值并在灰度阶段收集实际处理时长数据校准；OCR 阶段按批次而非整阶段设置超时；误杀后重试是幂等的不会产生脏数据，最坏情况为资源浪费。
- 进度持久化写入频率与存储方案未完全验证：tf-20 标注 evidence_strength=weak，500ms/10 chunk 的节流策略和数据库写入方案在 10+ 大文件并发场景下是否真能控制写入压力在基线 20% 以内，需在性能测试中验证。若不达标可能需要引入 Redis 缓存层或异步写入队列，增加方案复杂度。
- 状态表 schema migration 对存量任务的影响：新增多个字段（error_type/last_heartbeat_at/stage_progress 等）需要对存量任务记录做迁移；存量正在执行的解析任务可能处于旧的单一「parsing」状态，上线后这些任务可能无法被新的心跳检测识别。缓解措施：上线时先将存量 parsing 任务标记为 failed 并提示用户重新发起，或提供一次性迁移脚本将旧状态映射到新枚举值。
- 前端轮询模式下 API 并发压力：多个用户同时查看大量解析任务时，≤5s 的轮询间隔可能对状态查询 API 造成压力。缓解措施：API 侧支持批量查询（一次返回多个任务状态）、添加 ETag/304 缓存、或限制前端同时轮询的任务数量。
- 【rag/svr 核心模块】task_executor 是本次改造核心载体：需实现 8 阶段状态机流转与持久化、Worker 心跳定期写入（30s 间隔）、结构化异常捕获与 error_type 分类（oom/ocr_timeout/worker_crash/dependency_error）、可重试 vs 不可重试判定、指数退避重试调度（10s/30s/90s 最多 3 次）、重试前残留数据清理（先删后建）、进度节流批量写入（500ms/10 chunk）、动态超时阈值按文件大小配置。请确认现有 task_executor 架构是否能承载如此大范围的改造，以及是否有足够的测试覆盖。
- 【rag/nlp 核心模块】需支持稳定 chunk ID 生成（文档 hash + 偏移量）保证重试 upsert 幂等；embedding 阶段需支持进度回调与批次级超时；embedding 模型调用需捕获依赖服务异常并分类为 dependency_error。请确认嵌入管线当前是否已有进度回调机制，以及 chunk ID 生成方式的改造对已有索引数据的兼容性影响。
- 【deepdoc/parser 核心模块】extracting 阶段的格式解析器（PDF/DOCX/Excel）需支持进度回调与阶段心跳写入；大文件解析需捕获 OOM 异常并向上层透传 error_type=oom；unsupported_format 需在解析入口校验并分类为不可重试错误。请确认各格式解析器是否能安全捕获 OOM（Python OOM 通常不可恢复），以及进度回调在现有解析架构中的可行性。
- 【rag/flow uncertain】管线编排层可能需要协调多阶段流转与心跳写入的编排逻辑，具体改造范围取决于现有编排架构是否已在 task_executor 内完成阶段切换。请人工确认是否有独立编排入口需要同步改造，避免编排层与状态机层产生不一致。
- 【rag/graphraph uncertain】GraphRAG 场景的自动重试安全性依赖图谱幂等性，本期不在 scope 内但需明确策略：对 GraphRAG 任务禁用自动重试或增加跳过已完成子图的前置检查。请人工确认 GraphRAG 任务入口与 task_executor 的关系，以及禁用自动重试是否会导致 GraphRAG 任务在失败后无法恢复。
- 【待确认-阈值类】AC-41 的 200MB 文件大小上限、AC-42 的动态超时阈值具体数值（extracting 10min/ocr 15min/embedding 5min 等）、AC-43 的 95% 成功率指标均为工程决策，上游证据（全部 mock 来源）未提供历史性能基线或压测数据。请人工确认这些数值是否有线上监控数据或容量规划依据。
