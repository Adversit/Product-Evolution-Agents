# 技术笔记：上传与解析状态机

文档从上传到可问答通常经历：`uploaded → queued → parsing(分步: 提取/OCR/分块/嵌入/索引) → done | failed`。当前实现把中间过程压缩成单一「解析中」态，前端只能轮询一个布尔/枚举状态，导致用户无法判断「卡住」还是「在跑」。

改进方向：在 task_executor 侧把解析拆成可上报的阶段，并为每阶段写入进度（已处理 chunk / 总 chunk、当前步骤名、worker 心跳时间戳）。前端按阶段渲染进度条与步骤名；后端通过状态表或消息存储阶段进度，避免只靠日志。

关键点：需要一个明确的状态枚举与「最后心跳时间」字段，配合超时判定（超过阈值无心跳 → 标记疑似卡住）。状态机本身是成熟工程实践，落地成本中等，主要改动在解析任务队列与状态数据模型。
